# e2ee-key-server

Сервис **①** из плана «4 Rust-микросервиса» (приоритет №1). Реализует
X3DH-style сервер обмена prekey-bundles: клиенты публикуют сюда публичный
ключевой материал, а тот, кто хочет начать переписку, забирает bundle
собеседника и локально (на клиенте) вычисляет общий секрет.

**Сервер никогда не видит и не хранит ни одного приватного ключа и не
выполняет ни одной DH-операции.** Это чистое хранилище+верификатор
публичного материала.

## Что это НЕ делает (сознательно вне рамок этого шага)

- Не шифрует и не пересылает сами сообщения — этим занимается
  существующий Node/Socket.IO слой (после того как клиенты через этот
  сервис обменяются ключами, всё дальнейшее шифрование — на клиенте).
- Не реализует Double Ratchet / хранение сессионного состояния — это
  тоже полностью клиентская логика поверх результата X3DH.
- Не выполняет никакой криптографии в браузере — это сервис ④
  (WASM-модуль), отдельный шаг.
- Не прячет от себя самого, кто чей bundle запросил (см. «Threat model»
  ниже — это осознанно принято, не забыто).

## Threat model (зафиксирована в переписке до этого шага)

| Параметр | Значение | Следствие для этого сервиса |
|---|---|---|
| Адверсарий | L4 — до юридического принуждения включительно | Сервер обязан физически не иметь возможности отдать приватные ключи по запросу/повестке — он их никогда не получает |
| Доверие серверу | Ноль, «слепой ретранслятор» | Сервер не участвует в вычислении общих секретов, только хранит/раздаёт публичные байты |
| Топология | Только 1-на-1 | Bundle всегда привязан к одному user_id, без концепции комнат/групп |
| Эфемерность | Оба слоя | `DELETE /internal/v1/keys` для полного стирания; ротация SPK и расход OPK поддерживают forward secrecy |
| Mixnet/PIR | **Явно не требуется** (это L5/L6, вне модели) | Сервер намеренно видит связку «кто чей ключ запросил» — задокументировано как принятый риск, а не недосмотр |

## Криптографический дизайн

### Два identity-ключа вместо одного (XEdDSA)

Классический X3DH (Signal) использует один Curve25519-ключ и для DH, и
для подписи — через конвертацию Montgomery↔Edwards (XEdDSA). Здесь
вместо этого используются **два независимых ключа на пользователя**:

- `identity_signing_key` — Ed25519, только для подписи Signed PreKey;
- `identity_dh_key` — X25519, только для DH1/DH2 в X3DH.

Это осознанное отступление от «чистого» протокола ради простоты и
корректности: не нужен самодельный код конвертации точек между
кривыми — исторически частый источник тонких багов. Итоговые
security-свойства (аутентичность SPK, forward secrecy, deniability)
не страдают.

### Что сервер проверяет, а что должен проверять клиент

Сервер проверяет подпись Signed PreKey **при загрузке** (defense in
depth — отбрасывает заведомо битые bundle). Но так как сервер не
доверенная сторона (L4), это не заменяет обязательную повторную
проверку подписи **на стороне клиента**, получающего bundle, перед тем
как использовать SPK в X3DH. Это должно быть реализовано в сервисе ④.

### Отбраковка low-order точек

Все публичные ключи проверяются на all-zero (известная low-order точка
Curve25519). Это минимальная defense-in-depth проверка; полная защита
от small-subgroup атак — ответственность клиентского кода,
выполняющего сам DH.

## API (внутренний, только для Node-бэкенда)

Base URL: `http://127.0.0.1:7420` (см. `BIND_ADDR` в `.env`).

Все эндпоинты кроме `/healthz` требуют заголовки:
```
X-Internal-Secret: <INTERNAL_SHARED_SECRET>
X-User-Id: <числовой id уже аутентифицированного сессией пользователя>
```

Порт **не должен** быть проброшен наружу — только на loopback, только
для Node.

| Метод | Путь | Назначение |
|---|---|---|
| GET | `/healthz` | health-check, без авторизации |
| PUT | `/internal/v1/keys/identity` | зарегистрировать/заменить identity-ключи |
| PUT | `/internal/v1/keys/signed-prekey` | ротация Signed PreKey |
| POST | `/internal/v1/keys/one-time-prekeys` | пополнить пул OPK (батч) |
| GET | `/internal/v1/keys/one-time-prekeys/count` | сколько OPK осталось у себя |
| GET | `/internal/v1/keys/bundle/:target_user_id` | забрать bundle для старта сессии (атомарно расходует один OPK у target) |
| DELETE | `/internal/v1/keys` | стереть весь ключевой материал вызывающего |

Ключи и подписи передаются как base64 (`STANDARD` алфавит, с padding).

### Пример: регистрация identity-ключей

```
PUT /internal/v1/keys/identity
{
  "identity_signing_key": "<base64, 32 байта, Ed25519 pub>",
  "identity_dh_key": "<base64, 32 байта, X25519 pub>"
}
→ { "success": true }
```

### Пример: получение bundle

```
GET /internal/v1/keys/bundle/42
→ {
  "identity_signing_key": "...",
  "identity_dh_key": "...",
  "signed_prekey": { "key_id": 7, "public_key": "...", "signature": "..." },
  "one_time_prekey": { "key_id": 103, "public_key": "..." } | null
}
```
`404`, если у пользователя 42 ещё нет identity-ключей или SPK
(E2EE-онбординг не завершён). `one_time_prekey: null`, если пул исчерпан
— X3DH в этом случае деградирует на один DH-шаг (см. RFC/спеку X3DH),
сессия остаётся безопасной, но с чуть меньшей forward secrecy для
самого первого сообщения. Клиент должен воспринимать это как сигнал
срочно попросить получателя пополнить OPK.

## Интеграция с Node (`server.js`) — **пока не сделана, следующий шаг**

Этот сервис самодостаточен и собирается/тестируется отдельно, но чтобы
браузер мог им воспользоваться, `server.js` должен добавить slim-прокси
роуты, которые:
1. проверяют `req.session.userId` как обычно (существующий middleware);
2. форвардят запрос на `http://127.0.0.1:7420/internal/v1/keys/...`,
   подставляя `X-Internal-Secret` из своего `.env` и
   `X-User-Id: req.session.userId`;
3. отдают ответ клиенту как есть.

Пример одного роута (остальные пять — по аналогии):

```js
const KEY_SERVER_URL = process.env.KEY_SERVER_URL || 'http://127.0.0.1:7420';
const KEY_SERVER_SECRET = process.env.INTERNAL_KEY_SERVER_SECRET;

app.put('/api/keys/identity', requireAuth, async (req, res) => {
    const r = await fetch(`${KEY_SERVER_URL}/internal/v1/keys/identity`, {
        method: 'PUT',
        headers: {
            'Content-Type': 'application/json',
            'X-Internal-Secret': KEY_SERVER_SECRET,
            'X-User-Id': String(req.session.userId),
        },
        body: JSON.stringify(req.body),
    });
    res.status(r.status).json(await r.json());
});
```

`INTERNAL_KEY_SERVER_SECRET` в Node-`.env` должен быть тем же значением,
что `INTERNAL_SHARED_SECRET` в `.env` этого сервиса.

Это следующий конкретный шаг, если хотите продолжить прямо сейчас —
я его не делал в этом инкременте, чтобы явно разделить «новый Rust-код»
и «правки в существующем server.js» на отдельные проверяемые куски.

## Запуск локально

```bash
cp .env.example .env
# отредактировать DATABASE_URL и INTERNAL_SHARED_SECRET
cargo run
# таблицы создаются автоматически при старте (embedded migrations)
```

Проверка:
```bash
curl http://127.0.0.1:7420/healthz
# ok
```

## Тесты

```bash
cargo test
```
Юнит-тесты покрывают самую чувствительную часть — `crypto.rs`:
round-trip подписи, отклонение подделанного сообщения, отклонение
подписи чужим ключом, отклонение all-zero и некорректной длины ключа.

Интеграционных тестов с реальной БД пока нет — потребуют
`DATABASE_URL` в CI, следующий шаг при желании.

## Toolchain note

Проект собран и запинован под **rustc 1.75** (то, что доступно через
`apt` в песочнице). Если на вашей машине стоит более новый rustc
(что вероятно), можно ослабить часть точных пинов в `Cargo.lock` через
`cargo update` — но текущий `Cargo.lock` рабочий «из коробки» на любой
версии Rust ≥ 1.75.

## Известные ограничения V1 (осознанно, не баги)

- Один identity per пользователь (мультидевайс не поддержан).
- Нет rate-limiting на уровне этого сервиса — предполагается, что Node
  уже ограничивает частоту запросов на своих `/api/*` роутах
  (`express-rate-limit` уже используется в проекте).
- Нет метрик/observability кроме структурных логов через `tracing`.
- Cross-service coupling через общий Postgres — если когда-нибудь
  сервисы физически разъедутся по разным БД, `user_id` как «мягкая»
  внешняя ссылка продолжит работать без изменений схемы.
